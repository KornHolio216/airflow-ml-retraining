## Short Overview

This project is a simple machine learning retraining pipeline managed by Apache Airflow. It loads data from a CSV file, trains a `RandomForestClassifier`, validates the new model and decides whether it should replace the current production model.

Retraining is useful when a model may need to be updated with new data. Validation is important because a new model should not be promoted only because it was trained later. In this pipeline, the new model is promoted only when it passes the quality check and performs better than the current production model.

## Project Goal

The goal of this project is to practice a small MLOps workflow with Airflow and Docker.

The project focuses on:

- orchestrating ML tasks with Apache Airflow,
- automating a simple model retraining workflow,
- keeping archived model versions,
- maintaining one production model,
- validating a model before promotion,
- making deployment conditional instead of automatic.

## Pipeline Workflow

```text
load data -> train model -> validate model -> compare with production -> promote if better
```

The DAG is named `retrain_model_dag` and contains three main tasks:

1. `train_model` - loads CSV data, trains a `RandomForestClassifier` and saves the model with a timestamp in `models/archive/`.
2. `validate_model` - calculates validation metrics for the new model and saves a metrics report.
3. `compare_and_promote_model` - compares the new model with the current production model and promotes it only if it meets the required conditions.

Main directories:

- `dags/` - contains the Airflow DAG.
- `data/` - contains the input CSV file.
- `models/archive/` - stores archived model versions.
- `models/production/` - stores the current production model and its score.
- `reports/metrics/` - stores validation reports generated by the pipeline.
- `alerts/` - stores a simple alert file if the model does not meet the minimum quality threshold.
- `logs/` - stores Airflow runtime logs.

## Model Validation

The validation task calculates:

- accuracy
- precision
- recall
- F1-score
- confusion matrix

Each run saves a metrics report in:

```text
reports/metrics/metrics_<timestamp>.json
```

The promotion decision is mainly based on `accuracy`. The current minimum required accuracy is:

```text
MINIMUM_ACCURACY = 0.80
```

## Conditional Deployment

The new model can replace the production model only when:

1. its accuracy is at least `MINIMUM_ACCURACY`,
2. its accuracy is higher than the current production score.

If both conditions are met, the pipeline promotes the new model and updates:

```text
models/production/production_model.pkl
models/production/production_score.txt
models/production/production_metrics.json
```

If the new model does not meet the minimum quality threshold, the pipeline writes an alert to:

```text
alerts/model_performance_alert.txt
```

Generated models, metrics and alerts are runtime artifacts and should not be committed to the repository.

## How To Run

First, initialize Airflow:

```bash
docker compose up --build airflow-init
```

Then start the full environment:

```bash
docker compose up --build
```

The Airflow UI will be available at:

```text
http://localhost:8080
```

Login details:

```text
login: admin
password: admin
```

## How To Test The Pipeline Manually

1. Open the Airflow UI.
2. Find the DAG named `retrain_model_dag`.
3. Trigger the DAG manually.
4. Open Graph View and check the task order:

```text
train_model -> validate_model -> compare_and_promote_model
```

5. Check the `validate_model` logs for validation metrics.
6. Check the `compare_and_promote_model` logs for the promotion decision.
7. Check `reports/metrics/` for the generated JSON report.
8. Check `models/production/` if the model was promoted.

## Output

After a successful DAG run, archived models are saved in:

```text
models/archive/
```

When a model is promoted, the current production model and related files are stored in:

```text
models/production/production_model.pkl
models/production/production_score.txt
models/production/production_metrics.json
```

Validation reports are saved in:

```text
reports/metrics/
```

## Limitations

- The dataset is small and uses generic feature names.
- The current model does not include preprocessing for categorical data.
- The promotion decision is still mainly based on accuracy.
- There is no prediction API yet.
- There is no dashboard yet.
- There is no MLflow experiment tracking yet.
- This is a small local MLOps prototype, not a production system.
